# Cursor User Rules - 개인 원칙 요약

이 문서는 Cursor의 **User Rules(전역 규칙)**에 넣기 좋은 “개인 원칙 요약” 템플릿이다.  
프로젝트에 더 구체적인 규칙이 있으면 **프로젝트 규칙이 우선**이다.

## 출력/커뮤니케이션
- 답변은 **항상 한글**로 작성한다.
- 불확실하면 질문을 늘어놓기보다, **가정(Assumptions)을 명시**하고 진행한다(필요 시 즉시 수정).
- 독립적인 작업은 **병렬로 수행**해 속도를 최적화한다.

## 작업 방식(Agent-first)
- 복잡한 작업은 먼저 **계획**을 세우고(Plan Mode 포함), 큰 변경은 단계적으로 나눈다.
- 코드 작성/수정 직후에는 **리뷰(코드 품질/보안/일관성)**를 수행한다.

## 테스트(TDD 우선)
- 신규 기능/버그 수정은 **TDD(RED → GREEN → REFACTOR)**를 기본으로 한다.
- 가능한 경우 **커버리지 80%+**를 목표로 하되, 프로젝트 정책이 있으면 그 기준을 따른다.
- 테스트 실패 시, 테스트가 틀린 게 아니라면 **구현을 수정**한다.

## 보안/프라이버시(필수)
- 시크릿(API 키/토큰/비밀번호/JWT 등)은 **코드/문서/로그에 절대 포함하지 않는다**(예시는 마스킹).
- 출력/로그 공유 전 민감정보가 없는지 점검한다.
- 외부/사용자 입력은 **경계에서 검증**한다(스택에 맞는 검증 도구 사용).

## 코딩 스타일/품질
- **불변성 우선**: 가능한 한 in-place mutation을 피하고 새 객체를 만들어 반환한다.
- 예외적으로 mutation이 더 안전/효율적이라면 **왜 필요한지 한글 주석으로 의도**를 남기고, 변경 범위를 최소화한다.
- **작고 응집도 높은 코드**를 선호한다(함수는 짧게, 파일은 과도하게 커지면 분리).
- 디버그 출력(`print()`, `console.log`) 대신 **로깅**을 사용한다.

## Git(해당 시)
- 커밋 메시지는 **Conventional Commits**(`feat:`, `fix:`, `refactor:`, `docs:`, `test:` 등)를 따른다.
- 커밋은 **작고 목적이 명확하게** 유지한다.
- 커밋 전/PR 전에는 가능한 범위에서 **로컬 테스트를 실행**한다.
