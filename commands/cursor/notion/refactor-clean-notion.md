# `/refactor-clean` — 테스트로 검증하며 “죽은 코드(Dead Code)”를 안전하게 제거하는 커맨드

`/refactor-clean`은 코드베이스에서 **미사용 코드/파일/의존성**을 찾아 **안전하게 삭제**하고, 매 단계마다 **테스트로 회귀를 막는** 정리(클린업) 커맨드입니다.  
핵심은 “크게 리팩터링”이 아니라, **하나씩(atomic) 지우고 매번 테스트로 검증**하는 안전 루프에 있습니다.

# 한눈에 보기(요약)

### 역할

- 프로젝트 타입에 맞는 도구로 **죽은 코드 후보를 탐지**
- 결과를 SAFE/CAUTION/DANGER로 분류해 **삭제 우선순위**를 결정
- SAFE 항목을 **한 번에 하나씩 삭제 → 전체 테스트 재실행**으로 검증
- CAUTION/DANGER는 동적 참조/외부 소비자/엔트리포인트 위험을 먼저 점검
- 정리 이후 중복/불필요 래퍼/재수출(re-export)까지 추가 정돈

### 언제 쓰나

- 기능 개발 후 쌓인 **미사용 유틸/컴포넌트/파일**을 정리할 때
- 의존성이 늘어나 **unused dependency**를 줄이고 싶을 때
- 리팩터링 전, 먼저 **표면적/잡음(dead code)**을 제거해 변경 난이도를 낮추고 싶을 때

### 출력(결과물)

- 삭제한 항목(함수/파일/의존성) 수 요약
- 건너뛴 항목(불확실/테스트 실패) 목록
- 제거된 라인 수(대략)와 “전체 테스트 통과” 상태

# 어떻게 쓰나(1분 사용법)

1. 죽은 코드 분석 도구를 실행해 후보를 모읍니다.
2. 결과를 SAFE/CAUTION/DANGER로 분류합니다.
3. SAFE부터 **하나 삭제 → 전체 테스트 → 다음 항목** 루프로 진행합니다.
4. 테스트가 깨지면 즉시 되돌리고(rollback) 해당 항목은 스킵합니다.

# 이 커맨드가 하는 일(진행 흐름)

### 1) 죽은 코드 탐지(프로젝트별 도구)

- JS/TS
  - `npx knip` (unused exports/files/deps)
  - `npx depcheck` (unused npm deps)
  - `npx ts-prune` (unused TS exports)
- Python
  - `vulture <PY_APP_DIR>/`
- Go
  - `deadcode ./...`
- Rust
  - `cargo +nightly udeps`

> 참고: `<PY_APP_DIR>/`는 Python 앱/패키지 루트로 치환합니다(예: `src/`, `app/`, `your_package/`).

### 2) 결과 분류(SAFE/CAUTION/DANGER)

- **SAFE**: 미사용 유틸, 테스트 헬퍼, 내부 함수 등 → 비교적 안전하게 삭제 가능
- **CAUTION**: 컴포넌트, API 라우트, 미들웨어 등 → 동적 import/외부 소비자 가능성 점검 필요
- **DANGER**: 설정 파일, 엔트리포인트, 타입 정의 등 → 건드리기 전 정밀 조사 필요

### 3) 안전 삭제 루프(SAFE)

- 전체 테스트를 한 번 돌려 **기준선(baseline)** 확보
- SAFE 항목을 **하나 삭제**
- 전체 테스트 재실행
  - 실패: 즉시 되돌리고 해당 항목 스킵
  - 성공: 다음 항목으로 이동

### 4) CAUTION 항목 점검(삭제 전 필수 체크)

- 동적 import 사용 여부 탐색: `import()`, `require()`, `__import__`
- 문자열 참조 탐색: 라우트 이름/컴포넌트 이름이 설정에 박혀있는지
- 퍼블릭 API로 export 되는지(패키지 외부 소비자 존재 가능성)
- 배포/퍼블리시된 패키지라면 dependents(의존 프로젝트) 고려

### 5) 중복/무의미한 레이어 정리(선택)

- 80%+ 유사한 함수 병합
- 중복 타입 정의 통합
- 가치 없는 래퍼 함수 인라인
- 의미 없는 re-export 제거

# 실무 팁 / 주의사항(중요)

- 규칙 1: **테스트 없이 삭제하지 않기**
- 규칙 2: **한 번에 하나씩**(atomic) 지우기 — 롤백이 쉬워집니다
- 규칙 3: 확신이 없으면 **스킵**(죽은 코드가 남는 게 서비스 장애보다 낫습니다)
- 규칙 4: 클린업 중에는 **리팩터링을 섞지 않기**(관심사 분리)

# Quick Commands(실행 예)

# Python
