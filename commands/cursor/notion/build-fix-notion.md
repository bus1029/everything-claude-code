# `/build-fix` — 빌드/타입 에러를 “최소 변경”으로 한 개씩 안전하게 복구하는 커맨드

`/build-fix`는 프로젝트의 **빌드 시스템을 감지**한 뒤, 빌드/타입 에러를 **파일 경로 기준으로 묶고(그룹핑)**, **의존성 순서에 맞춰** “한 번에 하나씩” 고치면서 매번 다시 빌드로 검증하는 슬래시 커맨드입니다.  
핵심 가치는 리팩터링이 아니라 **작고 확실한 diff로 빌드를 다시 녹색으로 만드는 것**입니다.

# 한눈에 보기(요약)

### 역할

- 프로젝트 **빌드 도구 자동 감지** 및 빌드 커맨드 실행
- 에러를 **파일별로 그룹핑**하고 **의존성 순서**로 정렬
- “**1 에러 → 최소 수정 → 재빌드 검증**” 루프 수행
- 진행 상황(총 에러 수/남은 에러) 추적에 유리한 형태로 정리

### 언제 쓰나

- 빌드가 깨져서 **어디부터 고칠지** 흐름이 필요할 때
- TypeScript/Go/Rust/Java 등에서 **타입/컴파일 에러를 빠르게 복구**해야 할 때
- 리팩터링 없이 **안전한 핫픽스 수준**으로 build green이 목표일 때

### 출력(결과물)

- 사용한 **빌드 커맨드(감지 결과)**
- 수정한 에러 목록(파일 경로 포함)
- 남아있는 에러(있다면)
- 새로 생긴 에러(이상적으론 0)
- 해결되지 않은 이슈의 다음 액션 제안

# 어떻게 쓰나(1분 사용법)

1. 채팅에 `/build-fix`를 입력합니다.
2. 에이전트가 프로젝트 징표 파일을 보고 빌드 시스템을 감지해 빌드를 실행합니다.
3. 에러를 파일별로 정리한 뒤, **하나씩 최소 수정**으로 고치고 매번 빌드로 확인합니다.

# 이 커맨드가 하는 일(진행 흐름)

### 1) 빌드 시스템 감지

아래 같은 “징표 파일”을 기준으로 빌드 커맨드를 고릅니다.

- `package.json`(build script): `npm run build` 또는 `pnpm build`
- `tsconfig.json`(TS only): `npx tsc --noEmit`
- `Cargo.toml`: `cargo build 2>&1`
- `pom.xml`: `mvn compile`
- `build.gradle`: `./gradlew compileJava`
- `go.mod`: `go build ./...`
- `pyproject.toml`: `python -m py_compile` 또는 `mypy .`

### 2) 에러 파싱/정렬

- stderr를 수집하고
- **파일 경로별로 그룹핑**
- **의존성 순서(임포트/타입 → 로직)**로 정렬
- 진행 추적을 위해 총 에러 수를 집계

### 3) Fix 루프(한 번에 하나)

각 에러마다 아래를 반복합니다.

- 파일의 에러 주변 맥락을 읽고
- 원인을 진단한 뒤(누락 import, 타입 불일치, 문법 오류 등)
- **가장 작은 수정**으로 해결
- 곧바로 빌드를 재실행해 “해결 + 부작용 없음”을 확인

### 4) 가드레일(멈춰야 하는 경우)

아래 상황이면 “계속 밀어붙이기”보다 **중단 후 방향 전환**이 필요합니다.

- 고쳤더니 **에러가 더 늘어남**
- 같은 에러가 **3번 시도해도 그대로**
- 해결에 **아키텍처 변경**이 필요
- 원인이 **의존성 누락/설치 필요**(패키지 설치/모듈 정리 등)

# 실무 팁 / 주의사항(중요)

- 원칙: **리팩터링 금지**, build green이 목표(최소 diff).
- 우선순위: **import/타입/컴파일 에러 → 로직 에러** 순으로 고치는 게 안전합니다.
- 의존성 문제(모듈/패키지 누락)는 코드보다 **설치/설정**이 원인일 가능성이 큽니다.

