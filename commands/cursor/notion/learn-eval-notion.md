# `/learn-eval` — 세션에서 패턴을 추출하고 “품질 게이트 + 저장 위치 결정” 후 스킬로 저장하는 커맨드

`/learn-eval`은 세션 로그에서 재사용 가능한 패턴을 뽑아 `/learn`처럼 스킬로 저장하되, 저장 전에 **(1) 품질 자체평가(루브릭)**와 **(2) 저장 위치(전역 vs 프로젝트) 결정**을 반드시 거치도록 확장한 커맨드입니다.  
핵심은 “아무 패턴이나 저장”이 아니라, **재사용 가치가 있는지 점검하고, 올바른 장소에 저장하며, 사용자 확인 후에만 파일을 쓰는 것**입니다.

# 한눈에 보기(요약)

### 역할

- 세션에서 재사용 가능한 “패턴” 후보를 추출
- 가장 가치 있는 인사이트 1개를 고르고 스킬 초안을 작성
- 전역/프로젝트 중 **저장 위치를 결정**
- 루브릭으로 **자체 품질 평가**(점수/근거 포함)
- 점수가 낮으면 초안을 개선하고 재평가
- 최종적으로 **저장 경로 + 점수표 + 최종 초안**을 보여주고 사용자 확인을 기다림

### 언제 쓰나

- 디버깅/에러 해결 과정에서 “다음에도 쓸 만한” 재사용 패턴이 생겼을 때
- 특정 라이브러리/환경/워크플로에서 반복되는 시행착오를 문서화하고 싶을 때
- 팀/개인 생산성을 위해 “잘 되는 해결 루틴”을 스킬로 축적하고 싶을 때

### 출력(결과물)

- 제안 저장 경로(전역 또는 프로젝트)
- 품질 평가 점수표(각 항목 점수 + 근거 + 총점)
- 스킬 파일 초안(정해진 frontmatter + 본문 섹션)
- “이 경로에 저장해도 될까요?” 확인 요청(명시적 승인 대기)

# 어떻게 쓰나(1분 사용법)

1. `/learn-eval`을 실행합니다.
2. 에이전트가 세션에서 추출할 만한 패턴을 고릅니다.
3. 저장 위치와 품질 점수를 함께 제시합니다.
4. 사용자가 승인하면 해당 경로에 스킬 파일을 저장합니다.

# 이 커맨드가 하는 일(진행 흐름)

### 1) 무엇을 추출하나(대상)

- 에러 해결 패턴(원인 → 해결 → 재사용성)
- 디버깅 테크닉(비직관적 단계, 도구 조합)
- 워크어라운드(라이브러리 퀴크, API 제한, 버전 특이점)
- 프로젝트 특화 패턴(컨벤션, 아키텍처 결정, 통합 패턴)

### 2) 저장 위치 결정(전역 vs 프로젝트)

- 질문: “이 패턴이 다른 프로젝트에서도 유용한가?”
- 전역(User-level): `<HOME>/.cursor/skills/learned/`
  - 2개 이상 프로젝트에서 재사용 가능(쉘 호환, 디버깅 루틴, LLM API 특성 등)
- 프로젝트(Project-level): `<REPO>/.cursor/skills/learned/`
  - 특정 레포에만 의미 있는 지식(특정 설정 파일의 퀴크, 프로젝트 구조 결정 등)
- 애매하면 전역을 우선(전역 → 프로젝트로 옮기는 게 더 쉬움)

### 3) 스킬 초안 생성 포맷(고정)

```markdown
---
name: pattern-name
description: "Under 130 characters"
user-invocable: false
origin: auto-extracted
---

# [Descriptive Pattern Name]

**Extracted:** [Date]
**Context:** [Brief description of when this applies]

## Problem
[What problem this solves - be specific]

## Solution
[The pattern/technique/workaround - with code examples]

## When to Use
[Trigger conditions]
```

### 4) 품질 자체평가(루브릭)

아래 5개 차원을 각각 1–5로 점수화합니다.

- Specificity
- Actionability
- Scope Fit
- Non-redundancy
- Coverage

규칙:

- 어떤 차원이라도 1–2점이면, 초안을 개선하고 재평가해서 **모두 3점 이상**이 되도록 합니다.
- 점수표(Score + Rationale)와 총점을 함께 제시합니다.

### 5) 저장 전 사용자 확인(필수)

- 제안 저장 경로 + 점수표 + 최종 초안을 보여주고
- **명시적 승인**을 기다린 뒤에만 실제 파일을 저장합니다.

# 실무 팁 / 주의사항(중요)

- 사소한 수정(오타/단순 문법 오류)은 패턴으로 저장하지 않는 편이 좋습니다.
- “1회성 장애/일시적 이슈”보다 **반복될 가능성이 큰 패턴**에 집중하세요.
- 스킬은 “한 파일에 한 패턴”이 유지보수에 유리합니다.
