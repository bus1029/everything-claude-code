# `/go-test` — Go에서 TDD(RED→GREEN→REFACTOR)로 구현하고 커버리지 80%+를 확인하는 커맨드

`/go-test`는 Go 코드 작업을 **테스트 주도 개발(TDD)** 흐름으로 강제하는 슬래시 커맨드입니다.  
핵심은 “바로 구현”이 아니라 **테이블 기반(table-driven) 테스트를 먼저 쓰고(RED)**, 그 테스트를 통과시키는 **최소 구현(GREEN)**, 그리고 **리팩터링(REFACTOR)**을 반복하며 **커버리지 80%+**를 확인하는 것입니다.

# 한눈에 보기(요약)

### 역할

- 타입/인터페이스/시그니처를 먼저 잡고
- **테이블 기반 테스트**를 먼저 작성(RED)
- 테스트가 “올바른 이유로 실패”하는지 확인
- 최소 구현으로 통과(GREEN) → 개선(REFACTOR)
- `go test -cover` 등으로 **커버리지 목표(기본 80%+)** 확인

### 언제 쓰나

- 새 Go 함수를 구현할 때
- 기존 코드에 테스트를 추가해 커버리지를 올릴 때
- 버그 수정 시(실패 테스트부터 작성)
- 중요한 비즈니스 로직을 안전하게 만들 때
- Go TDD 패턴을 학습/정착시키고 싶을 때

### 출력(결과물)

- 작성된 테스트(테이블 기반 케이스)
- 테스트 실행 결과(RED → GREEN)
- 리팩터링 후에도 녹색 유지 확인
- 커버리지 수치 및 리포트(필요 시 profile)

# 어떻게 쓰나(1분 사용법)

1. 채팅에 `/go-test`와 함께 “만들고 싶은 동작”을 적습니다.
   - 예: `/go-test 이메일 검증 함수를 TDD로 만들어줘`
2. 에이전트가 함수 시그니처를 잡고, **테스트를 먼저** 작성합니다(RED).
3. 테스트를 실행해 실패를 확인한 뒤, 최소 구현으로 통과시킵니다(GREEN).
4. 리팩터링하고, 커버리지까지 확인합니다.

# 이 커맨드가 하는 일(진행 흐름)

### TDD 사이클

- **RED**: 실패하는 테스트(테이블 기반) 작성
- **GREEN**: 통과하는 최소 구현
- **REFACTOR**: 품질 개선(테스트는 계속 녹색)
- **REPEAT**: 다음 케이스 추가

### 권장 테스트 패턴(Go 관용구)

- 테이블 기반 테스트(`tests := []struct { ... }{ ... }`)
- 서브테스트(`t.Run`)
- 병렬 테스트 시 루프 변수 캡처(`tt := tt`) 후 `t.Parallel()`
- 헬퍼에는 `t.Helper()`, 정리에는 `t.Cleanup(...)`

# 실무 팁 / 주의사항(중요)

- 금지: 구현부터 쓰고 테스트를 나중에 붙이는 방식(RED 단계를 건너뛰면 TDD 효과가 급감합니다).
- 금지: 테스트에서 `time.Sleep`로 기다리기(플래키의 대표 원인).
- 커버리지 목표는 기본 80%+지만, **핵심 비즈니스 로직은 100%**를 목표로 두는 편이 안전합니다.

# Quick Commands(실행 예)

```bash
# 전체 테스트
go test ./...

# 기본 커버리지
go test -cover ./...

# 커버리지 프로파일 생성
go test -coverprofile=coverage.out ./...

# HTML로 확인
go tool cover -html=coverage.out

# 함수별 커버리지
go tool cover -func=coverage.out

# 레이스 + 커버리지
go test -race -cover ./...
```
